# 压力测试平台 — 接口设计

## 约定

### 请求规范

> **协议说明**：
> - **前端 ↔ Master**：HTTP 协议（`http_no_protocol`），端口 8080
> - **Worker ↔ Master 内部通信**：tRPC 协议，Master 端口 8081，Worker 端口 9090
>
> 本文档接口路径沿用 REST 风格描述，内部 tRPC 接口的方法名与路径一一对应。

- Base URL（前端调用，HTTP）：`http://master:8080/api`
- Base URL（Worker 调用，tRPC）：`trpc.stress.master.internal`，地址 `master:8081`
- 前端请求/响应均为 JSON，`Content-Type: application/json`
- 除登录接口外，所有前端接口需携带 `Authorization: Bearer <token>`

### 统一响应格式

```json
{
  "code":    0,
  "message": "success",
  "data":    {}
}
```

### 分页参数（GET 列表接口通用）

| 参数 | 类型 | 说明 |
|------|------|------|
| `page` | int | 页码，从 1 开始，默认 1 |
| `pageSize` | int | 每页条数，默认 20，最大 100 |

### 分页响应格式（data 字段）

```json
{
  "list":  [],
  "total": 100,
  "page":  1,
  "pageSize": 20
}
```

## 一、用户认证

> 支持两种登录方式：**账号密码登录**（§1.1）和**手机号验证码登录**（§1.2 + §1.3）。
> 两种方式均返回相同格式的 `token` 和用户信息，后续请求携带方式完全一致。

### 1.1 账号密码登录

```
POST /api/auth/login
```

**Request**
```json
{
  "username": "admin",
  "password": "123456"
}
```

**Response**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token":      "eyJhbGci...",
    "expires_at": 1711900800,
    "user": {
      "id":       1,
      "username": "admin",
      "email":    "admin@example.com",
      "role":     1
    }
  }
}
```

---

### 1.2 发送短信验证码

```
POST /api/auth/sms/send
```

**Request**
```json
{
  "phone": "13800138000"
}
```

**Response**
```json
{ "code": 0, "message": "success", "data": null }
```

> **限流规则**：同一手机号 60 秒内只允许发送一次，重复请求返回错误码 1004。
> 验证码有效期 5 分钟，6 位纯数字，由服务端生成后调用短信服务商发送。

**错误码**

| code | 含义 |
|------|------|
| 1003 | 手机号格式错误 |
| 1004 | 发送过于频繁，请 60 秒后重试 |
| 1005 | 短信服务商发送失败 |

---

### 1.3 手机号验证码登录

```
POST /api/auth/login/sms
```

**Request**
```json
{
  "phone": "13800138000",
  "code":  "123456"
}
```

**Response**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token":      "eyJhbGci...",
    "expires_at": 1711900800,
    "user": {
      "id":       1,
      "username": "admin",
      "phone":    "13800138000",
      "role":     1
    }
  }
}
```

> **登录逻辑**：根据 `phone` 查找用户，若用户不存在则拒绝登录（返回 1006）。
> 验证码校验通过后立即标记为已使用，不可二次使用。

**错误码**

| code | 含义 |
|------|------|
| 1003 | 手机号或验证码格式错误 |
| 1006 | 该手机号未绑定账号 |
| 1007 | 验证码错误或已过期 |

**Request**：无 body

## 五、场景配置

> 场景配置与任务 1:1 绑定，任务创建后可随时更新场景配置。

### 5.1 更新场景配置

```
PUT /api/tasks/{task_id}/scene
```

**Request**
```json
{
  "mode":      "step",
  "duration":  300,
  "timeoutMs": 5000,

  "targetRps":  500,
  "rpsRampTime": 20,
  "rpsMode":    "fixed",

  "steps": [
    { "concurrent": 50,  "rampTime": 20, "duration": 100 },
    { "concurrent": 100, "rampTime": 20, "duration": 100 },
    { "concurrent": 200, "rampTime": 30, "duration": 100 }
  ],
  "rpsSteps": [],

  "circuitBreaker": {
    "enabled":              true,
    "errorRateThreshold":   10.0,
    "windowSeconds":        30,
    "minRequests":          100
  }
}
```

> **`mode`**：`"step"`=VU 阶梯，`"rps"`=定速 RPS
>
> **`rpsMode`**（仅 mode="rps" 时有效）：`"fixed"`=固定速率，`"step"`=阶梯爬升
>
> **`rpsRampTime`**（仅 mode="rps", rpsMode="fixed" 时有效）：从 0 线性爬升到 `targetRps` 所需时间（秒），`0` = 立即起步
>
> **`steps`**：VU 阶梯配置数组（mode="step"）：`[{"concurrent": 100, "rampTime": 20, "duration": 60}, ...]`
>   - `rampTime`：从上一阶段线性爬升到本阶段并发数所需时间（秒），`0` 表示瞬变，第一阶段从 0 爬升到目标值
>
> **`rpsSteps`**：RPS 阶梯配置数组（mode="rps", rpsMode="step"）：`[{"rps": 100, "duration": 60, "rampTime": 0}, {"rps": 300, "duration": 60, "rampTime": 30}, ...]`
>   - `rampTime`：从上一阶段线性爬升到本阶段 rps 所需时间（秒），`0` 表示瞬变，第一阶段的 `rampTime` 表示从 0 爬升到目标值的时长

## 十五、Master 内部接口（Worker 调用，tRPC 协议）

> 以下接口运行在 Master 的 **8081 端口**，使用 **tRPC 协议**，service name 为 `trpc.stress.master.internal`。
> Worker 通过 trpc-go client 调用，仅限内部网络访问。

### 15.1 Worker 注册（由 Nacos 驱动，无独立接口）

> Worker 服务注册与发现通过 **Nacos** 完成，无需调用 Master 接口。
>
> Worker 启动时通过 `nacos-sdk-go v2` 向 Nacos 注册实例：
> - `serviceName`: `stress-worker`，`groupName`: `DEFAULT_GROUP`
> - `metadata`: `{ workerId, cpu_cores, mem_total_gb, max_concurrency }`
>
> Master 订阅 `stress-worker` 服务：
> - 实例**上线**事件 → 从 metadata 读取配置，写入 `worker_node(status=online)`
> - 实例**下线**事件 → 更新 `status=offline`，若有进行中的 run 则标记为 failed

---

### 15.2 Worker 负载心跳（每 5 秒，纯负载同步）

> **职责收窄**：存活探测由 Nacos 健康检查负责，此接口仅同步运行时负载数据，供 Master 调度评分使用。

```
POST /internal/workers/{worker_id}/heartbeat
```

**Request**
```json
{
  "cpuUsage":           45.2,
  "memUsage":           38.6,
  "maxConcurrency":     2000,
  "currentConcurrency": 200,
  "runningRunId":       "exec-1001"
}
```

> `maxConcurrency` 每次心跳均携带，Master 收到后直接覆盖写入 `worker_node.max_concurrency`，确保 Worker 配置动态变更后自动同步，无需重新注册。

**Response**
```json
{ "code": 0, "data": null }
```

### 15.4 Worker 上报接口级指标（Run 结束时，一次性）

> 每个 URL pattern 的 RT histogram 和错误计数，用于报告 Percentile 表格。
> 在 Run 状态变为 `success / stopped / circuit_broken` 后，Worker 触发此上报。

```
POST /internal/metrics/api-report
```

**Request**
```json
{
  "runId":    "exec-1001",
  "workerId": "uuid-xxx",
  "apis": [
    {
      "pattern":       "/api/goods/:id",
      "requests":      8234,
      "errors":        12,
      "rtHistogram": {
        "0":    45,
        "10":   1230,
        "50":   3812,
        "100":  2180,
        "200":  712,
        "500":  198,
        "1000": 45,
        "5000": 12
      }
    }
  ]
}
```

> `rtHistogram` 的 key 为桶下界（ms），value 为落入该桶的请求数量。
> 桶边界固定：`0, 10, 50, 100, 200, 500, 1000, 5000, +Inf`。
> Master 将多个 Worker 同一 pattern 的 histogram 对应桶累加，再重新计算分位值。

**Response**
```json
{ "code": 0, "data": null }
```
