# 压力测试平台 — 压测脚本体系设计

## 一、整体设计理念

参考 k6 的设计思路，将脚本体系分为四层：

```
┌──────────────────────────────────────────────────────────┐
│                   脚本 Git 仓库层                          │
│   独立 Git 仓库维护所有测试脚本，本地 IDE 开发，PR 审核      │
│   push 后 CI 自动编译，产物上传对象存储（MinIO / OSS）       │
└──────────────────────────┬───────────────────────────────┘
                           │ CI 通知 Master 新版本可用
┌──────────────────────────▼───────────────────────────────┐
│                     脚本规范层                             │
│   定义脚本必须遵守的接口约定（Setup / Teardown / Default）  │
│   用户按规范编写 Go 脚本                                    │
└──────────────────────────┬───────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────┐
│                     SDK 框架层                             │
│   平台提供给脚本调用的能力库                                │
│   sdk/http、sdk/check、sdk/vars、sdk/log、sdk/sleep       │
└──────────────────────────┬───────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────┐
│                     执行引擎层                             │
│   Worker 从对象存储拉取编译好的 .so，plugin.Open 加载执行   │
│   不在 Worker 侧编译，保证多节点产物一致                     │
└──────────────────────────────────────────────────────────┘
```

## 二、脚本 Git 工作流

### 2.1 整体流程

```
┌─────────────────────────────────────────────────────────────────┐
│                      scripts-repo（独立 Git 仓库）               │
│                                                                  │
│  scripts/                                                        │
│  ├── http_login/                                                 │
│  │   ├── main.go          ← 脚本主体（实现 ScriptEntry）          │
│  │   └── main_test.go     ← 单元测试（可本地 go test 验证逻辑）    │
│  ├── grpc_order/                                                 │
│  │   └── main.go                                                 │
│  └── cache_profile/                                              │
│      └── main.go                                                 │
│                                                                  │
│  go.mod  ← module: stress-scripts
│           require github.com/Aodongq1n/jarvan4-platform/sdk v0.x.x│
└────────────────────────────┬────────────────────────────────────┘
                             │ git push / PR merge
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                    CI（GitHub Actions / Jenkins）                 │
│                                                                  │
│  1. go vet / staticcheck（质量门禁，失败则阻断）                   │
│  2. go test ./...（跑单测）                                       │
│  3. 遍历 scripts/ 下每个子目录：                                  │
│     go build -buildmode=plugin \                                 │
│       -o dist/{script_name}/{commit_hash}.so \                   │
│       ./scripts/{script_name}                                    │
│  4. 上传到对象存储：                                              │
│     scripts/{script_name}/{commit_hash}.so                       │
│  5. 通知 Master：POST /internal/scripts/publish                  │
│     { name, commit_hash, artifact_url, commit_message, author } │
└────────────────────────────┬────────────────────────────────────┘
                             │ 通知 Master 新版本可用
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                         Master                                   │
│                                                                  │
│  写入 script 表（记录版本、artifact_url、commit 信息）             │
│  前端脚本列表页实时刷新，可选择版本绑定到压测任务                    │
└────────────────────────────┬────────────────────────────────────┘
                             │ 下发任务时携带 artifact_url
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                         Worker × N                               │
│                                                                  │
│  1. 从对象存储按 artifact_url 下载 .so（本地缓存，避免重复拉取）    │
│  2. plugin.Open(".so")                                           │
│  3. plugin.Lookup("Script") → ScriptEntry 接口                  │
│  4. 调度引擎执行 Setup / Default / Teardown                      │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 编译环境一致性

Go Plugin 的 `.so` 有严格的编译环境要求：**Go 版本、OS、依赖版本必须与 Worker 运行环境完全一致**，否则 `plugin.Open` 会报 `plugin was built with a different version of package` 错误。

**解决方案**：CI 使用与 Worker 相同的 Docker 镜像编译。

```dockerfile
# ci/builder.Dockerfile  与 worker/Dockerfile 使用相同基础镜像
FROM golang:1.22-bookworm

WORKDIR /build
COPY . .
RUN go build -buildmode=plugin -o dist/script.so ./scripts/http_login
```

```yaml
# .github/workflows/build.yml
jobs:
  build:
    runs-on: ubuntu-latest
    container:
      image: golang:1.22-bookworm   # ← 与 Worker 部署镜像保持一致
    steps:
      - uses: actions/checkout@v4
      - name: Build plugins
        run: |
          for dir in scripts/*/; do
            name=$(basename "$dir")
            go build -buildmode=plugin \
              -o "dist/${name}/${GITHUB_SHA}.so" \
              "./${dir}"
          done
      - name: Upload to MinIO
        run: |
          for f in dist/**/*.so; do
            mc cp "$f" "minio/stress-scripts/$f"
          done
      - name: Notify Master
        run: |
          curl -X POST "$MASTER_URL/internal/scripts/publish" \
            -H "Content-Type: application/json" \
            -d "{\"commit_hash\":\"$GITHUB_SHA\", ...}"
```

### 2.3 对象存储路径规范

```
Bucket: stress-scripts
路径:   scripts/{script_name}/{commit_hash}.so

示例:
  scripts/http_login/a3f8c1d2e4b5.so
  scripts/grpc_order/a3f8c1d2e4b5.so
  scripts/cache_profile/b7e9f2a1c3d6.so
```

Worker 本地缓存目录：`/var/cache/stress-worker/scripts/{script_name}/{commit_hash}.so`，命中缓存则跳过下载。

### 2.4 SDK 升级时的全量重编译机制

Go Plugin 对依赖版本有严格要求：**`stress-platform/sdk` 的任何变更（哪怕是新增一个函数）都会导致所有历史 `.so` 与新版 Worker 不兼容**，`plugin.Open` 会报 `plugin was built with a different version of package` 错误。

**决策：SDK 升级时强制触发所有脚本重新 CI 编译。**

这个"强制触发"本身需要一个明确的入口，有两种方式可选：

| 方式 | 实现 | 适用场景 |
|------|------|---------|
| **自动触发**（跨仓库） | SDK 仓库 CI 完成后，通过 GitHub Actions `repository_dispatch` 事件自动触发 `scripts-repo` 的全量重建 workflow | 团队规模较大、脚本仓库变更频繁 |
| **手动触发**（半自动） | SDK 升级时，由发布者手动在 `scripts-repo` 触发一次 `workflow_dispatch`（GitHub Actions 手动触发），并在 SDK PR 描述中注明"需触发脚本全量重建" | 内部工具、SDK 升级不频繁 |

**本平台采用手动触发方式**（内部工具场景，SDK 升级频率低，流程简单优先）。

`jarvan4-platform` 仓库的 CI 通过路径过滤，仅当 `sdk/` 目录有变更时才发出跨仓库通知（或提示手动触发）：

```yaml
# jarvan4-platform/.github/workflows/sdk-release.yml
# 用途：SDK 目录变更时提醒发布者触发 scripts-repo 全量重编译
name: SDK Change Detected

on:
  push:
    branches: [main]
    paths:
      - 'sdk/**'   # 只有 sdk/ 目录有变更才触发，master/worker 变更不触发

jobs:
  notify:
    runs-on: ubuntu-latest
    steps:
      - name: Remind to rebuild scripts
        run: |
          echo "⚠️  SDK 有变更，请在 scripts-repo 手动触发 Rebuild All Scripts workflow"
          echo "触发地址：https://github.com/Aodongq1n/stress-scripts/actions/workflows/rebuild-all.yml"
          # 如需自动触发，取消注释以下步骤并配置 PAT：
          # curl -X POST \
          #   -H "Authorization: token ${{ secrets.SCRIPTS_REPO_PAT }}" \
          #   -H "Accept: application/vnd.github.v3+json" \
          #   https://api.github.com/repos/Aodongq1n/stress-scripts/dispatches \
          #   -d '{"event_type":"sdk-updated"}'
```

```yaml
# scripts-repo/.github/workflows/rebuild-all.yml
# 用途：SDK 升级后手动触发全量重编译
name: Rebuild All Scripts

on:
  workflow_dispatch:
    inputs:
      reason:
        description: '触发原因（如：SDK v1.2.0 升级）'
        required: true

jobs:
  rebuild:
    runs-on: ubuntu-latest
    container:
      image: golang:1.22-bookworm   # 必须与 Worker 镜像一致
    steps:
      - uses: actions/checkout@v4
      - name: Force update SDK
        run: go get github.com/Aodongq1n/jarvan4-platform/sdk@latest && go mod tidy
      - name: Test
        run: go test ./... -timeout 60s -count=1
      - name: Build all plugins
        run: |
          for dir in scripts/*/; do
            name=$(basename "$dir")
            go build -buildmode=plugin \
              -o "dist/${name}/${GITHUB_SHA}.so" \
              "./${dir}"
          done
      - name: Upload & Notify Master
        run: |
          # 上传所有 .so，逐一通知 Master 新版本可用
          for dir in scripts/*/; do
            name=$(basename "$dir")
            # mc cp / aws s3 cp ...
            curl -X POST "$MASTER_URL/internal/scripts/publish" \
              -H "Content-Type: application/json" \
              -d "{\"name\":\"${name}\",\"commit_hash\":\"${GITHUB_SHA}\", ...}"
          done
```

**操作规范**：
- SDK 发版打 tag 必须加目录前缀：`git tag sdk/v0.2.0 && git push origin sdk/v0.2.0`（Go 多模块约定，不加前缀 `go get` 无法解析子模块版本）
- SDK 发版 PR 的描述中必须注明是否存在 breaking change（影响 `.so` 兼容性）
- 合并 SDK PR 后，发布者在 `scripts-repo` 手动触发 `Rebuild All Scripts` workflow，填写触发原因
- 全量重编译完成前，**不应升级生产环境 Worker 镜像**（新 Worker + 旧 `.so` 会导致加载失败）
- Worker 滚动升级顺序：① 全量重编译完成 → ② 滚动更新 Worker Pod

### 2.5 与原"页面编辑脚本"方案的对比

| 维度 | 页面编辑（原方案） | Git 工作流（现方案） |
|------|-------------------|---------------------|
| 开发体验 | 浏览器 textarea，无补全 | 本地 IDE，完整 Go 补全 + 类型检查 |
| 错误发现时机 | 运行时 Worker 编译失败 | push 前本地 `go test`，CI 编译门禁 |
| 版本管理 | 数据库版本号 | Git commit hash，完整历史追溯 |
| 团队协作 | 无 | PR Review、分支、blame |
| 多 Worker 一致性 | 每个 Worker 各自编译，理论上一致 | 统一产物，所有 Worker 下载同一个 .so |
| 依赖管理 | 禁止外部依赖（Worker 编译时无法解析） | 同样禁止，go.mod 只允许 `github.com/Aodongq1n/jarvan4-platform/sdk` |

## 三、脚本规范（Script Spec）

### 3.1 k6 对照参考

| k6 概念 | 本平台对应 | 说明 |
|---------|-----------|------|
| `export default function` | `Default(ctx)` | 每次压测迭代执行的主体 |
| `export function setup()` | `Setup(ctx)` | 压测开始前执行一次，返回共享数据 |
| `export function teardown()` | `Teardown(ctx, data)` | 压测结束后执行一次，清理资源 |
| `options` | `ScriptOptions()` | 可选：脚本自带默认场景参数 |
| `check()` | `sdk.Check()` | 断言验证 |
| `http.get/post` | `sdk.HTTP.Get/Post` | HTTP 请求 |

### 3.2 脚本必须实现的接口

```go
// 所有压测脚本必须实现此接口
// 文件：sdk/spec/script.go（平台提供，脚本 import）
package spec

// ScriptEntry 是每个压测脚本必须实现并导出的接口
// 导出符号名固定为："Script"
type ScriptEntry interface {
    // Setup 在压测开始前执行一次（全局，非每个goroutine）
    // 返回的 data 会传递给每个 goroutine 的 Default 和最终的 Teardown
    // 可以在这里做：初始化连接池、预加载测试数据等
    // 若无需前置逻辑，实现为空方法返回 nil 即可
    // ctx 可通过 ctx.Vars.Env() 读取平台下发的环境变量
    Setup(ctx *RunContext) (data interface{}, err error)

    // Default 是压测的核心逻辑，每次迭代调用一次
    // ctx 携带：vars（变量上下文）、setupData、迭代信息
    // 该方法的每次调用对应一次"虚拟用户"的行为
    Default(ctx *RunContext) error

    // Teardown 在压测全部结束后执行一次
    // data 是 Setup 的返回值
    // 可以在这里做：清理数据、关闭连接等
    Teardown(ctx *RunContext, data interface{}) error
}
```

### 3.3 RunContext：脚本的运行时上下文

```go
// 文件：sdk/spec/context.go（平台提供）
package spec

import (
    "context"
    "time"
)

// RunContext 是 Setup / Default / Teardown 三个生命周期方法统一使用的上下文
// Setup 阶段：用于读取环境变量、初始化连接池
// Default 阶段：用于执行迭代逻辑、记录指标
// Teardown 阶段：用于清理资源
type RunContext struct {
    context.Context

    // VU 信息（Setup/Teardown 阶段 VUId=0，Iteration=0）
    VUId      int    // 当前虚拟用户编号（goroutine 编号），从 1 开始
    Iteration int64  // 当前迭代次数（本 VU 已执行的次数）
    WorkerID  string // 执行本脚本的 Worker 节点 ID

    // 变量上下文（goroutine 级别隔离，VU 间不共享）
    // Env(key) 读取 Master 下发的平台环境变量（只读）
    // Set/Get 读写 VU 私有变量
    Vars VarStore

    // Setup 阶段返回的共享数据（只读，所有 VU 共享同一份）
    // Default/Teardown 阶段可用，Setup 阶段为 nil
    SetupData interface{}

    // HTTP 客户端（最常用协议，作为快捷方式内置）
    // 内部自动调用 Recorder 上报指标，脚本无需手动记录
    HTTP  HTTPClient

    // 断言
    Check Checker

    // 日志（上报到 Master 日志流，可在实时看板查看）
    Log   Logger

    // 睡眠（可被引擎停止信号中断，优先于 time.Sleep）
    Sleep Sleeper

    // 协议无关的指标记录器
    // HTTP/gRPC/Redis 等 SDK 函数内部自动调用，通常脚本无需直接使用
    // 仅在使用平台未内置的协议时，手动调用记录自定义指标：
    //   ctx.Recorder.Record("redis.GET", duration, err)
    Recorder MetricsRecorder
}

// MetricsRecorder 协议无关的指标上报接口
// 所有协议 SDK 内部统一通过此接口上报，Worker 聚合后推送给 Master
type MetricsRecorder interface {
    // Record 记录一次操作的指标
    //   label    - 操作标识，建议格式："协议.操作"，如 "redis.GET"、"/order.OrderService/Create"
    //   duration - 操作耗时
    //   err      - nil 表示成功，非 nil 计入失败
    Record(label string, duration time.Duration, err error)

    // Skip 标记最近一次 Record 不计入统计（如限流场景）
    Skip()
}
```

### 4.2 HTTP 客户端 SDK

```go
// sdk/http/client.go
package http

// HTTPClient HTTP 请求客户端接口
type HTTPClient interface {
    Get(url string, opts ...RequestOption) (*Response, error)
    Post(url string, body interface{}, opts ...RequestOption) (*Response, error)
    Put(url string, body interface{}, opts ...RequestOption) (*Response, error)
    Delete(url string, opts ...RequestOption) (*Response, error)
    Do(req *Request) (*Response, error)
}

// RequestOption 请求选项
type RequestOption func(*Request)

func WithHeader(key, value string) RequestOption { ... }
func WithHeaders(headers map[string]string) RequestOption { ... }
func WithQuery(key, value string) RequestOption { ... }
func WithTimeout(ms int) RequestOption { ... }
func WithBasicAuth(user, pass string) RequestOption { ... }

// WithName 显式指定接口 pattern，用于报告中的 Percentile 表格归类。
// 不指定时 Worker 会自动将路径中的纯数字和 UUID 替换为 :id / :uuid；
// 但对于非数字的动态段（如 slug、hash）必须手动指定，否则每个值都会成为独立 pattern。
//
// 示例：
//   ctx.HTTP.Get("/api/goods/"+goodsID, sdkhttp.WithName("/api/goods/:id"))
func WithName(pattern string) RequestOption { ... }

// Response HTTP 响应
type Response struct {
    StatusCode int
    Headers    map[string][]string
    Body       []byte
    Duration   time.Duration  // 请求耗时，平台自动记录

    // 平台内部使用：是否计入指标统计
    // 脚本可调用 res.Skip() 标记本次请求不计入统计（如限流跳过）
    skipped bool
}

func (r *Response) JSON(path string) interface{}  { ... } // JSONPath 提取
func (r *Response) Text() string                  { ... }
func (r *Response) Skip()                         { r.skipped = true }
```

### 4.3 多协议客户端 SDK

所有协议客户端遵循相同的设计模式：**连接/客户端在 `Setup` 里初始化，通过 `setupData` 传给 `Default` 使用；SDK 函数内部自动调用 `ctx.Recorder` 上报指标，脚本无需手动记录。**

#### gRPC

```go
// sdk/grpc/client.go
package grpc

// Conn gRPC 连接（在 Setup 中创建，所有 VU 共享）
type Conn struct { ... }

// Dial 建立 gRPC 连接
func Dial(ctx *spec.RunContext, target string, opts ...DialOption) (*Conn, error) { ... }

// Invoke 发起 unary gRPC 调用
// 内部自动调用 ctx.Recorder.Record(method, duration, err)
// label 格式："/package.Service/Method"，与 gRPC method 一致
func Invoke(ctx *spec.RunContext, conn *Conn, method string, req interface{}, opts ...CallOption) (*Response, error) { ... }

type Response struct {
    Status  codes.Code
    Body    []byte // proto 序列化后的字节，脚本用 proto.Unmarshal 解析
}
```

#### Redis

```go
package redis
```

#### Kafka


### 4.5 ScriptError（结构化错误类型）

脚本的 `Default()` 方法可以直接 `return err`（普通 Go error），Worker 会将其归类为系统错误；
若需要向平台上报**业务错误码**（被测服务返回的应用层错误），应 `return &spec.ScriptError{...}`。

```go
// sdk/spec/error.go
package spec

// ScriptError 脚本可返回的结构化错误。
// Worker 调用 errors.As(err, &ScriptError{}) 识别该类型；
// 无法解析的普通 error 回退为 system 错误，Code 填充原始 error.Error() 字符串。
type ScriptError struct {
    // Type 错误分类：
    //   "business" — 被测服务返回的业务错误（如余额不足、库存不足），由脚本解析后填入
    //   "system"   — 网络超时、连接拒绝、协议层错误，通常由 SDK 内部产生
    Type string // "business" | "system"

    // Code 错误码，字符串形式，平台透传不解释语义。
    //   业务错误：被测服务定义的业务错误码，如 "10001"、"ORDER_FAIL"
    //   系统错误：标准化标识，如 "TIMEOUT"、"CONNECTION_REFUSED"、"DECODE_ERROR"
    Code string

    // Message 错误描述，对应被测服务的 message 字段或 error.Error() 原始字符串。
    // 平台按 (Type, Code) 聚合计数，Message 取同 Code 下第一条写入报告。
    Message string

    // API 可选，发生错误的接口 path pattern。
    // 填入后报告"接口维度指标"中该接口的错误率可精确归因；
    // 不填时 Worker 尝试从当前 HTTP 请求上下文中自动推断。
    API string // 可选，如 "/v1/order/create"

    // LatencyMs 可选，发生错误时的请求耗时（ms）。
    // 填入后错误请求也会被计入 RT histogram，使分位值更准确。
    LatencyMs int // 可选
}

func (e *ScriptError) Error() string {
    return fmt.Sprintf("[%s] %s: %s", e.Type, e.Code, e.Message)
}
```

**使用示例**：

```go
func (s *OrderScript) Default(ctx *spec.RunContext) error {
    res, err := ctx.HTTP.Post(baseURL+"/api/order/create", body)
    if err != nil {
        // 网络层错误，直接 return，Worker 归类为 system 错误
        return err
    }

    // 解析业务响应
    var resp struct {
        Code    int    `json:"code"`
        Message string `json:"message"`
    }
    if err := json.Unmarshal(res.Body, &resp); err != nil {
        return &spec.ScriptError{
            Type: "system", Code: "DECODE_ERROR",
            Message: err.Error(), LatencyMs: int(res.Duration.Milliseconds()),
        }
    }
    if resp.Code != 0 {
        return &spec.ScriptError{
            Type: "business", Code: strconv.Itoa(resp.Code),
            Message:   resp.Message,
            API:       "/api/order/create",
            LatencyMs: int(res.Duration.Milliseconds()),
        }
    }
    ctx.Check.That(res).Status(200).RTLt(800)
    return nil
}
```

### 5.2 完整的链路脚本（登录 → 查询 → 下单）

```go
// script: 电商下单链路
package main

import (
    "encoding/json"
    "fmt"
    "math/rand"
    spec "github.com/Aodongq1n/jarvan4-platform/sdk/spec"
    sdkhttp "github.com/Aodongq1n/jarvan4-platform/sdk/http"
)

var Script = &OrderFlowScript{}

type OrderFlowScript struct{}

// setupData 是 Setup 阶段返回的共享数据，所有 VU 只读访问
type setupData struct {
    GoodsIDs []int64  // 预加载的商品 ID 列表
}

// Setup 压测开始前执行一次：预加载商品列表
func (s *OrderFlowScript) Setup(ctx *spec.RunContext) (interface{}, error) {
    res, err := ctx.HTTP.Get(
        ctx.Vars.Env("BASE_URL") + "/api/goods/list?page=1&size=100",
    )
    if err != nil {
        return nil, fmt.Errorf("setup failed: %w", err)
    }
    if res.StatusCode != 200 {
        return nil, fmt.Errorf("setup: goods list status %d", res.StatusCode)
    }

    var ids []int64
    items := res.JSON("data.list").([]interface{})
    for _, item := range items {
        m := item.(map[string]interface{})
        ids = append(ids, int64(m["id"].(float64)))
    }
    return &setupData{GoodsIDs: ids}, nil
}

// Default 每次迭代：登录 → 查询商品 → 下单
func (s *OrderFlowScript) Default(ctx *spec.RunContext) error {
    baseURL := ctx.Vars.Env("BASE_URL")

    // Step 1: 登录（仅首次迭代执行，后续复用 token）
    if ctx.Vars.GetString("token") == "" {
        res, err := ctx.HTTP.Post(baseURL+"/api/user/login",
            map[string]string{
                "username": fmt.Sprintf("test_user_%d", ctx.VUId),
                "password": "123456",
            },
            sdkhttp.WithHeader("Content-Type", "application/json"),
        )
        if err != nil {
            return err
        }
        ok := ctx.Check.That(res).Status(200).BodyContains("token")
        if !ok {
            return nil // 断言失败，已计入 fail，直接返回
        }
        ctx.Vars.Set("token", res.JSON("data.token"))
        ctx.Vars.Set("user_id", res.JSON("data.user_id"))
    }

    token  := ctx.Vars.GetString("token")
    userID := ctx.Vars.Get("user_id")

    // Step 2: 随机选一个商品
    data := ctx.SetupData.(*setupData)
    goodsID := data.GoodsIDs[rand.Intn(len(data.GoodsIDs))]

    // Step 3: 查询商品详情
    res, err := ctx.HTTP.Get(
        fmt.Sprintf("%s/api/goods/%d", baseURL, goodsID),
        sdkhttp.WithHeader("Authorization", "Bearer "+token),
    )
    if err != nil {
        return err
    }
    ctx.Check.That(res).Status(200).RTLt(300)

    // Step 4: 下单（带自定义业务签名）
    timestamp := time.Now().UnixMilli()
    appSecret := ctx.Vars.Env("APP_SECRET")
    sign := fmt.Sprintf("%s_%d_%d", appSecret, timestamp, goodsID) // 自定义签名逻辑

    body, _ := json.Marshal(map[string]interface{}{
        "user_id":   userID,
        "goods_id":  goodsID,
        "quantity":  1,
        "timestamp": timestamp,
        "sign":      sign,
    })

    orderRes, err := ctx.HTTP.Post(
        baseURL+"/api/order/create",
        body,
        sdkhttp.WithHeader("Authorization", "Bearer "+token),
        sdkhttp.WithHeader("Content-Type", "application/json"),
    )
    if err != nil {
        return err
    }

    // 限流不算失败
    if orderRes.StatusCode == 429 {
        ctx.Log.Warn("VU[%d] 触发限流，本次跳过", ctx.VUId)
        orderRes.Skip() // 不计入成功/失败统计
        return nil
    }

    ctx.Check.That(orderRes).Status(200).BodyContains("order_id").RTLt(800)
    return nil
}

// Teardown 压测结束后清理（可选）
func (s *OrderFlowScript) Teardown(ctx *spec.RunContext, data interface{}) error {
    ctx.Log.Info("压测结束，共加载 %d 个商品", len(data.(*setupData).GoodsIDs))
    return nil
}
```

### 5.3 配合阶梯加压的缓存接口脚本

> **说明**：阶梯加压是**执行引擎的行为**，由场景配置中的 `mode=2`（VU 阶梯）或
> `mode=3 + rps_mode=2`（RPS 阶梯爬升）及 `steps_json` 数组参数驱动 Worker
> 逐步增加 VU goroutine 数量或线性提升 RPS 速率。**脚本本身感知不到压力模式**，
> 也不应包含任何阶梯逻辑。
>
> 本示例的重点在于**脚本侧配合缓存场景的两个关键写法**：
> 1. 用 `ctx.VUId % N` 分散缓存 key，避免所有请求命中同一个 key
> 2. 用 `RTLt(100)` 对缓存接口设置严格的 RT 断言（缓存接口 > 100ms 视为异常）
>
> 实际阶梯配置示例（在任务场景配置中设置，与脚本无关）：
> ```
> mode=2, step_start=10, step_increment=10, step_interval=30, step_max=200, duration=600
> 效果：从 10 并发起步，每 30 秒增加 10 个 VU，最终压到 200 并发，共压测 10 分钟
> ```

```go
// script: 用户 Profile 缓存接口压测（适合配合阶梯加压场景使用）
package main

import (
    "fmt"
    spec "github.com/Aodongq1n/jarvan4-platform/sdk/spec"
    sdkhttp "github.com/Aodongq1n/jarvan4-platform/sdk/http"
)

var Script = &CacheStressScript{}

type CacheStressScript struct{}

func (s *CacheStressScript) Setup(ctx *spec.RunContext) (interface{}, error) {
    return nil, nil
}

func (s *CacheStressScript) Default(ctx *spec.RunContext) error {
    // 用 VUId 取模，模拟不同用户请求，避免全部命中同一缓存 key
    // 10000 个不同 key 足以覆盖缓存热数据场景
    userID := ctx.VUId % 10000

    res, err := ctx.HTTP.Get(
        fmt.Sprintf("%s/api/user/profile/%d", ctx.Vars.Env("BASE_URL"), userID),
        sdkhttp.WithHeader("Authorization", "Bearer "+ctx.Vars.Env("FIXED_TOKEN")),
    )
    if err != nil {
        return err
    }
    // 缓存接口严格要求 RT < 100ms，超出即视为性能劣化
    ctx.Check.That(res).StatusIn(200, 304).RTLt(100)
    return nil
}

func (s *CacheStressScript) Teardown(ctx *spec.RunContext, data interface{}) error {
    return nil
}
```

## 六、Worker 执行引擎设计

### 6.1 脚本执行机制：Go Plugin

```
CI 编译产物上传到对象存储
       │ artifact_url 存入 script 表
       ▼
Master 下发任务时携带 artifact_url（不再下发源码）
       │
       ▼
Worker 收到任务
       │
       ├─ 1. 检查本地缓存：/var/cache/stress-worker/scripts/{name}/{hash}.so
       │      命中缓存 → 直接跳到步骤 3
       │
       ├─ 2. 从对象存储下载 .so
       │      GET s3://stress-scripts/scripts/{name}/{hash}.so
       │      保存到本地缓存目录
       │
       ├─ 3. plugin.Open(".so")
       │
       ├─ 4. plugin.Lookup("Script") → ScriptEntry 接口
       │
       └─ 5. 调度引擎按模式执行 Script.Setup / Default / Teardown
```

### 6.2 Worker 引擎工程结构

```
worker/internal/
├── handler/
│   └── task.go
├── engine/
│   ├── runner.go           # 主调度器，管理 goroutine 生命周期
│   ├── mode_step.go        # VU 阶梯：按 steps_json 逐阶段增加 goroutine（含爬坡）
│   ├── mode_rps.go         # 定速 RPS：令牌桶控速（固定速率 & 阶梯爬升子模式）
```

```go
// script_runner.go 伪代码
func (r *ScriptRunner) runOnce(ctx *RunContext) {
    start := time.Now()

    err := r.script.Default(ctx)   // 执行脚本

    duration := time.Since(start)

    // 统计逻辑
    for _, res := range ctx.collectedResponses {  // 收集本次迭代内所有请求
        if res.skipped {
            continue  // Skip() 标记的请求不计入
        }
        if res.assertFailed || err != nil {
            r.metrics.RecordFail(res.Duration, res.failReason)
        } else {
            r.metrics.RecordSuccess(res.Duration)
        }
    }
}
```

```go
// shared/dto/task.go

// SubTaskConfig Master 下发给 Worker 的子任务配置
type SubTaskConfig struct {
    RunID    int64  `json:"run_id"`
    WorkerID string `json:"worker_id"`

    // 压测模式配置
    Mode       int `json:"mode"`        // 1=VU阶梯 2=定速RPS
    Duration   int `json:"duration"`    // 压测总时长(秒)
    TimeoutMs  int `json:"timeout_ms"`  // 单次请求超时(ms)

    // 定速RPS模式（mode=2）
    RPS     int `json:"rps"`      // 目标RPS（RPS固定子模式）
    RpsMode int `json:"rps_mode"` // 1=固定速率 2=阶梯爬升

    // 阶梯配置（mode=1 VU阶梯 或 mode=2 RPS阶梯时有效）
    // VU阶梯：每个元素为 VUStep；RPS阶梯：每个元素为 RPSStep
    Steps []interface{} `json:"steps"`

    // 环境变量（来自绑定的 environment，注入到 ctx.Vars.Env()）
    EnvVars map[string]string `json:"env_vars"`

    // 压测脚本列表（按 weight 比例混合执行）
    Scripts []ScriptItem `json:"scripts"`

    // Master 内部接口地址（Worker 上报指标用）
    ReportAddr string `json:"report_addr"`
}

// VUStep VU阶梯模式的单阶段配置（mode=1）
type VUStep struct {
    Concurrent int `json:"concurrent"` // 目标并发数
    RampTime   int `json:"ramp_time"`  // 从上一阶段线性爬升所需时间(秒)，0=瞬变，第一阶段从0起步
    Duration   int `json:"duration"`   // 稳定持续时长(秒)
}

// RPSStep RPS阶梯模式的单阶段配置（mode=3, rps_mode=2）
type RPSStep struct {
    RPS      int `json:"rps"`       // 目标RPS
    Duration int `json:"duration"`  // 稳定持续时长(秒)，不含爬坡
    RampTime int `json:"ramp_time"` // 从上一阶段线性爬升到本阶段RPS的时长(秒)，0表示瞬变
}

// ScriptItem 单个脚本项
type ScriptItem struct {
    ScriptID    int64             `json:"script_id"`
    ScriptName  string            `json:"script_name"`
    CommitHash  string            `json:"commit_hash"`  // Git commit hash，Worker 用于本地缓存 key
    ArtifactURL string            `json:"artifact_url"` // 对象存储 .so 路径，Worker 按此拉取
    Lang        string            `json:"lang"`         // "go"（预留扩展）
    Weight      int               `json:"weight"`       // 执行权重（前置脚本忽略此字段）
    EnvVars     map[string]string `json:"env_vars"`     // 脚本级环境变量，Worker 在启动脚本时注入
                                                        // 脚本通过 ctx.Vars.Env(key) 读取
                                                        // 典型用例：{"INTERFACE_WEIGHTS":"{\"search\":70,\"order\":30}"}
}
```

## 九、脚本编写规范（用户文档）

### 9.1 约定

| 约定 | 说明 |
|------|------|
| package 必须为 `main` | Go plugin 要求 |
| 必须导出 `var Script` 变量 | Worker 通过 `plugin.Lookup("Script")` 获取 |
| `Script` 必须实现 `spec.ScriptEntry` 接口 | 平台约定 |
| 不允许启动独立 goroutine | 并发由平台控制，脚本内禁止 `go func()` |
| 不允许 `os.Exit()` | 会导致 Worker 进程退出 |
| Default() 中的 panic 会被引擎 recover | 计入 fail，Worker 不会崩溃 |
| 脚本内 import 路径只允许标准库 + `github.com/Aodongq1n/jarvan4-platform/sdk` | 不允许引入外部依赖 |
| 每个脚本放在 `scripts/{script_name}/` 独立子目录 | CI 按子目录遍历编译 |
| 脚本名（子目录名）一旦确定不要修改 | 与平台 `script.name` 字段绑定 |

### 9.2 可用 SDK 列表

| SDK | Import 路径 | 说明 |
|-----|------------|------|
| HTTP 客户端 | `github.com/Aodongq1n/jarvan4-platform/sdk/http` | 发起 HTTP/HTTPS 请求，自动上报指标 |
| gRPC 客户端 | `github.com/Aodongq1n/jarvan4-platform/sdk/grpc` | 发起 gRPC Unary 调用，自动上报指标 |
| Redis 客户端 | `github.com/Aodongq1n/jarvan4-platform/sdk/redis` | Redis 读写操作，自动上报指标 |
| Kafka 客户端 | `github.com/Aodongq1n/jarvan4-platform/sdk/kafka` | Kafka 生产/消费，自动上报指标 |
| WebSocket 客户端 | `github.com/Aodongq1n/jarvan4-platform/sdk/ws` | WebSocket 连接/收发，自动上报指标 |
| TCP 客户端 | `github.com/Aodongq1n/jarvan4-platform/sdk/tcp` | 原生 TCP，适合自定义二进制协议 |
| 断言 | 通过 `ctx.Check` 访问 | HTTP/gRPC 结果断言 |
| 变量存储 | 通过 `ctx.Vars` 访问 | goroutine 级别变量 KV 存储，`Env()` 读平台环境变量 |
| 指标记录 | 通过 `ctx.Recorder` 访问 | 协议无关指标上报（各 SDK 内部已自动调用，一般无需手动使用） |
| 日志 | 通过 `ctx.Log` 访问 | 日志输出（会上报到平台） |
| 睡眠 | 通过 `ctx.Sleep` 访问 | 迭代间隔（可被平台中断信号响应） |
| 随机工具 | `github.com/Aodongq1n/jarvan4-platform/sdk/random` | 随机字符串、随机整数等 |

### 9.3 扩展语言支持预留

```
lang 字段预留：
  "go" → 当前支持，Go plugin 机制执行
  "js" → 后续扩展，goja 引擎执行（JS 不需要编译，更轻量）

Worker 的 script_loader.go 按 lang 字段分发到不同的 Loader 实现：
  GoLoader  → 编译 + plugin.Open
  JSLoader  → goja.New() + 注入 SDK
```

## 十、脚本可测试性设计

### 11.1 为什么天然可测试

脚本的 `Default(ctx *RunContext)` 所有外部依赖均为接口，不是具体实现：

```
ctx.HTTP    → HTTPClient 接口
ctx.Check   → Checker    接口
ctx.Vars    → VarStore   接口
ctx.Log     → Logger     接口
ctx.Sleep   → Sleeper    接口
```

测试时只需注入 mock 实现，**无需真实网络、无需 Worker 环境、无需编译为 .so**，标准 `go test` 即可运行。

### 11.3 testkit 核心实现

**testkit/context.go** — 可控的 RunContext 工厂：

```go
package testkit

import "github.com/Aodongq1n/jarvan4-platform/sdk/spec"

type Option func(*builder)

type builder struct {
    http    spec.HTTPClient
    checker spec.Checker
    vars    spec.VarStore
    logger  spec.Logger
    vuId    int
    iter    int64
}

func NewContext(opts ...Option) *spec.RunContext {
    b := &builder{
        http:    &MockHTTPClient{},
        checker: &MockChecker{},
        vars:    NewMockVarStore(nil),
        logger:  &MockLogger{},
    }
    for _, o := range opts {
        o(b)
    }
    return &spec.RunContext{
        VUId:      b.vuId,
        Iteration: b.iter,
        HTTP:      b.http,
        Check:     b.checker,
        Vars:      b.vars,
        Log:       b.logger,
    }
}

func WithHTTP(c spec.HTTPClient) Option       { return func(b *builder) { b.http = c } }
func WithVars(vars map[string]string) Option  { return func(b *builder) { b.vars = NewMockVarStore(vars) } }
func WithVUId(id int) Option                  { return func(b *builder) { b.vuId = id } }
func WithIteration(i int64) Option            { return func(b *builder) { b.iter = i } }
```

**testkit/mock_http.go** — 按 URL 预设响应，记录请求供断言：

```go
package testkit

import sdkhttp "github.com/Aodongq1n/jarvan4-platform/sdk/http"

type MockHTTPClient struct {
    // 按 URL 预设响应，未命中则返回 200 空 body
    Responses map[string]*sdkhttp.Response
    // 记录所有发出的请求，供测试断言
    Requests  []*sdkhttp.Request
}

func (m *MockHTTPClient) Do(req *sdkhttp.Request) (*sdkhttp.Response, error) {
    m.Requests = append(m.Requests, req)
    if resp, ok := m.Responses[req.URL]; ok {
        return resp, nil
    }
    return &sdkhttp.Response{StatusCode: 200, Body: []byte(`{}`)}, nil
}

// Get/Post/Put/Delete 内部均调用 Do
func (m *MockHTTPClient) Get(url string, opts ...sdkhttp.RequestOption) (*sdkhttp.Response, error) {
    return m.Do(&sdkhttp.Request{URL: url, Method: "GET"})
}
func (m *MockHTTPClient) Post(url string, body interface{}, opts ...sdkhttp.RequestOption) (*sdkhttp.Response, error) {
    return m.Do(&sdkhttp.Request{URL: url, Method: "POST", Body: body})
}
```

```go
func TestDefault_LoginSuccess(t *testing.T) {

        },
    }

    ctx := testkit.NewContext(

    assert.Len(t, mockHTTP.Requests, 1)
    assert.Equal(t, "POST", mockHTTP.Requests[0].Method)
```

### 11.5 CI 流水线里的测试集成

```yaml
# .github/workflows/ci.yml
jobs:
  test-and-build:
    runs-on: ubuntu-latest
    container:
      image: golang:1.22-bookworm   # 与 Worker 部署镜像保持一致

    steps:
      - uses: actions/checkout@v4

      - name: Lint
        run: go vet ./... && staticcheck ./...

      - name: Test               # 在编译 .so 之前先跑单测，失败即阻断
        run: go test ./... -v -timeout 60s -count=1

      - name: Build plugins      # 测试通过后才编译
        run: |
          for dir in scripts/*/; do
            name=$(basename "$dir")
            go build -buildmode=plugin \
              -o "dist/${name}/${GITHUB_SHA}.so" \
              "./${dir}"
          done

      - name: Upload & Notify
        run: |
          # 上传 .so 到对象存储，通知 Master
          ...
```

### 11.6 测试边界说明

| 测试类型 | 能否在 `go test` 里跑 | 说明 |
|----------|----------------------|------|
| 脚本业务逻辑（Default/Setup/Teardown） | ✅ | 注入 mock RunContext，完整覆盖 |
| 编译合法性 | ✅ | `go vet` + `go build`（不用 buildmode=plugin） |
| 断言逻辑、变量操作、错误处理 | ✅ | mock HTTPClient 模拟任意响应 |
| `.so` 加载后的集成验证（plugin.Open） | ❌ | `go test` 不支持 `-buildmode=plugin`，只能在 Worker 环境里验证 |

> `.so` 加载的集成验证由平台的「脚本调试」功能（单次试跑，§2.4）覆盖，两者分工明确：**单测覆盖业务逻辑，调试功能覆盖加载集成**。
