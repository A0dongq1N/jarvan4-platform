### 分层依赖规则

```
api  →  app  →  domain  ←  infra
```

- **domain** 不依赖任何外部包（无 GORM、无 Redis、无 tRPC）
- **infra** 实现 domain 定义的接口，依赖方向向内
- **app** 只依赖 domain 接口，不感知 infra 的具体实现（通过构造函数注入）
- **api** 只依赖 app 的 service，通过 DTO 与 domain entity 隔离

### 充血 vs 贫血的选择原则

不追求全面充血，按实际复杂度决策：

**充血（行为方法放在实体上）**：适用于有真实状态机或一致性约束的实体

| 实体 | 充血的原因 |
|------|-----------|
| `TaskRun` | 状态流转复杂（pending→running→success/stopped/failed/circuit_broken），快照写入后不可修改，必须由实体自己守护不变式 |
| `Task` | `BindScript` / `UnbindScript` 需要防重复绑定，`UpdateScene` 需要校验模式合法性 |
| `WorkerNode` | `Offline()` / `UpdateLoad()` 有明确的状态语义，`LoadScore()` 是领域计算逻辑 |

**贫血（普通 struct，逻辑在应用层）**：适用于以查询/CRUD 为主、无复杂状态机的实体

`User`、`Project`、`Script`、`Report`、`AuditLog` — 强行充血只会增加代码量，收益不足

**值对象（不可变 struct，无身份标识）**：

`ScriptSnapshot`、`WorkerSnapshot`、`StepConfig`、`RpsStepConfig`、`CircuitBreakerConfig` — 这些天然不可变，用 Go 的 struct 值语义表达，写入后不应被修改

```go
// 充血示例：TaskRun 状态机由实体自身保证，外部无法绕过
func (r *TaskRun) Start(workers []WorkerSnapshot, scripts []ScriptSnapshot) error {
    if r.status != StatusPending {
        return domain.ErrInvalidStateTransition
    }
    r.workerSnapshots = workers  // 快照写入，此后不可变
    r.scriptSnapshots = scripts
    r.status = StatusRunning
    return nil
}

func (r *TaskRun) CircuitBreak(reason string) error {
    if r.status != StatusRunning {
        return domain.ErrInvalidStateTransition
    }
    r.status = StatusCircuitBroken
    r.errorMsg = reason
    return nil
}

// 应用层只做编排，不含业务规则
func (s *ExecutionService) StartExecution(ctx context.Context, taskId string) (*TaskRun, error) {
    run, _     := s.runRepo.FindById(ctx, taskId)
    workers    := s.scheduler.SelectWorkers(...)
    snapshots  := s.buildScriptSnapshots(...)
    run.Start(workers, snapshots)    // 状态校验和快照绑定在实体内完成
    s.runRepo.Save(ctx, run)
    s.workerClient.DispatchTask(...)
    return run, nil
}
```

## 二、领域设计

### 2.1 限界上下文与子域划分

#### 限界上下文

平台整体是**一个限界上下文**，不做多 BC 拆分。原因：单团队、单体部署、领域概念无歧义（`Task`/`Script` 在任何地方含义相同）。强行拆分只会引入跨 BC 数据同步和防腐层的额外负担。

**唯一的物理边界：Worker 是独立 BC**

Worker 服务满足独立限界上下文的全部特征：独立部署、无持久化存储、独立领域模型（Engine/Collector/Reporter）。两个 BC 之间通过 `shared/dto/` 的 `SubTaskConfig` 和 `MetricsPayload` 通信，这就是 Context Map 的数据契约，**不允许在 Master 内部直接复用这两个 DTO**，否则打破边界。

#### 子域划分

| 子域类型 | 子域 | 说明 |
|----------|------|------|
| **核心子域** | 执行引擎（Execution） | Worker 调度、状态机、快照、降级——平台的核心竞争力，值得最精细的设计和最高的测试覆盖 |
| **支撑子域** | 任务配置（Task + Scene）| 没有它执行引擎无法工作，但自身逻辑不复杂 |
| **支撑子域** | 脚本管理（Script）| CI 发布驱动，平台侧以查询为主 |
| **支撑子域** | 压测报告（Report）| 核心价值之一，独立性强，不影响执行链路 |
| **通用子域** | 身份认证（Auth/User）| 通用能力，可替换为第三方服务 |
| **通用子域** | 项目管理（Project）| 多租户隔离，纯 CRUD |
| **通用子域** | 审计日志（Audit）| 横切关注点，通过中间件自动注入，对业务无侵入 |

> **设计投入策略**：核心子域精心设计、充血建模、多写测试；支撑子域适度设计；通用子域够用即止。

### 2.2 实体与行为方法

#### TaskRun（核心子域，充血实体）

状态机：

```
PENDING ──Start()──► RUNNING ──Complete()────► SUCCESS
                        │
                        ├──Stop()─────────────► STOPPED
                        ├──CircuitBreak()─────► CIRCUIT_BROKEN
                        └──Fail()─────────────► FAILED
```

字段约束：

| 字段 | 可变性 |
|------|--------|
| `id` / `taskId` / `triggerType` / `triggeredBy` | 创建后不可变 |
| `status` | 只能通过行为方法改变 |
| `scriptSnapshots` / `workerSnapshots` / `warningMsg` / `startTime` | `Start()` 写入后不可变 |
| `endTime` / `errorMsg` | 终态方法写入后不可变 |
| `initSteps` | 只能通过 `UpdateInitStep()` 追加/更新单步状态，不可整体替换 |

行为方法签名：

```go
// 工厂函数
func NewTaskRun(taskId, triggeredBy string, triggerType int) *TaskRun

// 状态转换
func (r *TaskRun) Start(workers []WorkerSnapshot, scripts []ScriptSnapshot, warningMsg string) error
func (r *TaskRun) Complete() error
func (r *TaskRun) Stop() error
func (r *TaskRun) Fail(msg string) error
func (r *TaskRun) CircuitBreak(reason string) error

// pending 阶段初始化步骤推进
func (r *TaskRun) UpdateInitStep(key string, status InitStepStatus, detail string) error

// 查询（只读访问器）
func (r *TaskRun) Status() Status
func (r *TaskRun) IsTerminated() bool          // success/stopped/failed/circuit_broken
func (r *TaskRun) ScriptSnapshots() []ScriptSnapshot
func (r *TaskRun) WorkerSnapshots() []WorkerSnapshot
```

#### Task（支撑子域，部分充血）

```go
func NewTask(projectId, name, description, createdBy string) *Task

func (t *Task) BindScript(scriptId, scriptName string, weight int) error
// 约束：同一 scriptId 不可重复绑定

func (t *Task) UnbindScript(scriptId string) error
// 约束：scriptId 不存在时返回 ErrNotFound

func (t *Task) UpdateScriptWeight(scriptId string, weight int) error
// 约束：weight 必须 > 0

func (t *Task) UpdateScene(cfg SceneConfig) error
// 约束：mode 必须是已知值；step 模式下 steps 不可为空

func (t *Task) Scripts() []TaskScript
func (t *Task) SceneConfig() SceneConfig
```

#### WorkerNode（支撑子域，部分充血）

```go
func (w *WorkerNode) UpdateLoad(cpuUsage, memUsage float64, concurrent int, runningRunId string)
// 心跳时调用，直接覆盖，无约束

func (w *WorkerNode) Offline() error
// 约束：已是 offline 状态时返回 ErrInvalidStateTransition

func (w *WorkerNode) LoadScore() float64
// 纯计算：cpuUsage×0.6 + memUsage×0.4，无副作用

func (w *WorkerNode) IsAvailable() bool
// status == online && runningRunId == ""
```

### 2.3 值对象清单

| 值对象 | 所属子域 | 不可变原因 |
|--------|----------|-----------|
| `ScriptSnapshot` | Execution | 执行时版本快照，写入即归档，不应被修改 |
| `WorkerSnapshot` | Execution | 同上 |
| `InitStep` | Execution | 通过 `UpdateInitStep()` 受控更新，不可整体替换 |
| `StepConfig` | Task | 描述一个压测阶段的参数，修改即为新阶段 |
| `RpsStepConfig` | Task | 同上 |
| `CircuitBreakerConfig` | Task | 熔断参数，整体替换，无局部修改语义 |
| `TaskScript` | Task | 绑定关系，通过 Task 行为方法管理，不单独修改 |

## 四、Redis 数据结构详细设计

```
# 执行状态
STRING  run:{runId}:status
  值：pending / running / success / stopped / failed / circuit_broken
  TTL：24h

# 最新实时指标（前端轮询直接读这个）
STRING  run:{runId}:metrics:latest
  值：JSON 序列化的 MetricsPoint
  TTL：24h，每秒覆盖写入

# 时序指标（ZSet，score = unix timestamp）
ZSET    run:{runId}:metrics:ts
  member：JSON(MetricsPoint)，score：unix timestamp
  TTL：24h，压测结束后持久化到 MySQL 再清除

# Worker 心跳（存活探测由 Nacos 负责，Redis 只存负载快照）
HASH    worker:{workerId}:load
  field cpu_usage            → "45.2"
  field mem_usage            → "38.6"
  field current_concurrency  → "200"
  field running_run_id       → "1001"
  TTL：15s（业务心跳每5秒续期，超15秒过期说明 Worker 已离线）
```

## 五、核心流程详细设计

### 4.1 启动压测流程

```
前端                    Master                              Worker
 │                        │                                   │
 │─── POST /executions ──►│                                   │
 │                        │ 1. 创建 task_run(pending)          │
 │                        │ 2. Worker 调度（见 §4.6）           │
 │                        │    a. 估算所需 Worker 数 N          │

 │                        │    d. 原子锁定（Redis SETNX）        │
 │                        │    e. 不足 N 台 → 降级 + 写 warning │
 │                        │ 3. 快照脚本版本 + Worker 信息        │

 │◄── 返回 executionId ───│                                   │
 │                        │                                   │ 5. 启动发压引擎
 │                        │◄── 每秒 POST /metrics/report ─────│    (全局指标，每秒)
 │                        │ 6. 聚合所有 Worker 全局指标          │
 │                        │ 7. 写入 Redis latest + ZSet         │
 │─── GET /executions/{id}/metrics（每1秒轮询）──────────────►│
 │◄── 返回最新指标 ─────────│                                   │
 │                        │                                   │ 8. Run 结束
 │                        │◄── POST /metrics/api-report ──────│    (接口级 histogram，一次性)
 │                        │ 9. 合并各 Worker histogram          │
 │                        │ 10.写入 MySQL report_percentiles   │
```

### 4.2 指标采集的两个维度

Worker 侧维护两套独立的采集结构，互不干扰：

```
┌─────────────────────────────────────────────────────────────┐
│ 维度一：全局滑动窗口（collector/global_metrics.go）           │
│  按秒滚动，每秒上报一次                                       │
│  统计量：QPS、并发数、总成功/失败数、平均 RT、P99 RT、错误分类  │
│  P99 用全局 histogram 分桶（无需保留原始样本）                  │
│  用途：实时看板（前端每秒轮询）                                │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 维度二：接口级 histogram（collector/api_metrics.go）          │
│  按 URL pattern 分组，在内存中持续累积，不清除                  │
│  统计量：每个 pattern 的请求数、错误数、RT histogram           │
│  Run 结束（success/stopped/circuit_broken）时一次性上报        │
│  用途：报告中的 Percentile 表格                               │
└─────────────────────────────────────────────────────────────┘
```

**URL pattern 提取规则**（避免路径参数导致的基数爆炸）：

脚本调用 HTTP SDK 时可通过 `WithName()` 选项显式指定 pattern：

```go
// 脚本侧：显式命名，避免 /api/goods/1、/api/goods/2 被视为不同接口
ctx.HTTP.Get(url, sdkhttp.WithName("/api/goods/:id"))
```

若脚本未传 `WithName`，Worker 按以下规则自动归一化：
1. 路径段纯数字 → 替换为 `:id`（`/order/12345` → `/order/:id`）
2. 路径段 UUID 格式 → 替换为 `:uuid`
3. Query String 整体丢弃（不作为 pattern 区分依据）

### 4.3 全局指标聚合逻辑

Master 每秒收到各 Worker 上报后进行聚合（全局维度）：

```
总 QPS          = Sum(worker.rps)
总成功数         = Sum(worker.successCount)
总失败数         = Sum(worker.failedCount)
总并发数         = Sum(worker.concurrent)
平均 RT         = Sum(worker.avgResponseTime × worker.total) / Sum(worker.total)
P99 RT          = 合并各 Worker 的 RT histogram 后重新计算分位值
错误率          = 总失败数 / (总成功数 + 总失败数) × 100%
```

### 4.4 接口级指标聚合逻辑

Run 结束后，Master 收到各 Worker 的接口级上报，按 pattern 合并：

```
同一 pattern，多 Worker 合并：
  total_requests  = Sum(worker.requests)
  total_errors    = Sum(worker.errors)
  error_rate      = total_errors / total_requests
  RT histogram    = 各 Worker histogram 对应桶计数相加
  P50/P75/P90/P95/P99/Max/Min = 从合并后的 histogram 重新计算分位值
```

结果写入 `report_percentiles` 表（见 §三 数据库设计）。

### 4.5 主动上报 + 主动拉取双模式

```
【Push 模式 - 主路径】
全局指标：Worker → 每秒 POST /internal/metrics/report → Master 写 Redis
接口级指标：Worker → Run 结束时 POST /internal/metrics/api-report → Master 聚合后写 MySQL

【Pull 模式 - 兜底/校验】
Master 调度器 → 每5秒 tRPC GetMetricsSnapshot → Worker 返回最新快照
用途：Push 超时/丢失时补偿；节点状态巡检
```

### 4.6 停止压测流程

```
前端 → POST /runs/{id}/stop → Master
  Master:
    1. 向所有参与的 Worker 发送 POST /worker/task/stop
    2. 更新 Redis status → "stopped"
    3. 更新 MySQL task_run.status → "stopped"
    4. 触发报告生成（异步 goroutine）

熔断检测（Master aggregator 每次聚合后执行）:
  条件：scene_config.cb_enabled=1 且 run.status=running
  算法：
    1. 从 Redis ZSet run:{runId}:metrics:ts 取最近 cb_window_sec 秒的数据点
    2. 汇总窗口内 success_count 和 fail_count
    3. 若 total < cb_min_requests → 跳过（冷启动保护）
    4. 若 fail_count/total*100 > cb_error_rate → 触发熔断：
       a. 向所有 Worker 发送 StopTask
       b. 更新 Redis status → "circuit_broken"
       c. 更新 MySQL task_run.status → "circuit_broken"，记录触发时的 error_msg（错误率快照）
       d. 触发报告生成（异步 goroutine）

报告生成（异步）:
    1. 从 Redis ZSet 读取所有时序数据
    2. 计算汇总指标（全局 P99 等）
    3. 写入 MySQL report + report_metrics
    4. 等待各 Worker 上报接口级指标（/internal/metrics/api-report）
    5. 合并 histogram → 写入 MySQL report_percentiles
    6. 清理 Redis 数据
```

### 4.7 Worker 注册、发现与心跳

两件事职责分离，由不同机制负责：

```
【服务注册与发现 — Nacos 负责】
Worker 启动:
  1. 读取本机 IP / Port(9090) / CPU 核心数 / 总内存 / max_concurrency
  2. 生成 UUID 作为 workerId
  3. 通过 nacos-sdk-go v2 向 Nacos 注册实例：
       serviceName: "stress-worker"
       group:       "DEFAULT_GROUP"
       ip:          <本机 IP>
       port:        9090
       metadata:    { workerId, cpu_cores, mem_total_gb, max_concurrency }
  4. Nacos Client 自动维护存活心跳（默认 5s），Master 无需感知

Worker 正常退出:
  调用 nacos client DeregisterInstance 主动注销，Nacos 立即通知 Master

Worker 异常宕机:
  Nacos 健康检查超时（默认 15s）后自动摘除实例，通知 Master

Master 订阅:
  启动时调用 nacos client Subscribe("stress-worker")
  监听 OnEvent 回调：
    实例上线 → 从 metadata 读配置 → 写 worker_node(status=online)
    实例下线 → 更新 worker_node(status=offline)
               → 若 running_run_id != "" → 标记 run 为 failed

【负载同步 — Worker → Master 业务心跳负责】
Worker 每 5 秒执行:
  POST /internal/workers/{workerId}/heartbeat
  携带: cpu_usage, mem_usage, current_concurrency, running_run_id

Master 收到负载心跳:
  1. 写 Redis worker:{workerId}:load（TTL=15s，供调度评分实时读取）
  2. 更新 MySQL worker_node.last_heartbeat / cpu_usage / mem_usage

Master 离线兜底（Nacos 已覆盖主路径，此为双保险）:
  定时任务每 30s 扫描 worker_node 中
  last_heartbeat < now()-30s 且 status=online 的节点
  → 更新 status=offline（防止 Nacos 事件丢失时节点僵尸留存）
```

### 4.8 Worker 调度策略

#### 需要几台 Worker

本质是资源估算，输入为场景配置，输出为所需 Worker 数量 N。

**VU 阶梯模式**：由峰值并发决定。
```
峰值并发 = steps[-1].concurrent（最后一阶段的目标并发数）
N = ⌈ 峰值并发 / 单Worker可用容量 ⌉
单Worker可用容量 = max_concurrency - current_concurrency
```

**RPS 模式**：由目标 RPS 决定。
```
单Worker额定RPS = max_concurrency / 预估平均RT(s)
  （保守估算：预估平均RT取 500ms，即 RT = 0.5s）
单Worker额定RPS ≈ max_concurrency * 2
N = ⌈ targetRps / 单Worker额定RPS ⌉
```

用户也可在启动压测时通过 `workerCount` 参数**手动指定**台数，跳过自动估算。手动指定优先级高于自动估算。

#### 选哪些 Worker（调度流程）

```
1. 【过滤】从 Redis worker:{id}:load 读取最新负载，筛选满足以下条件的节点：
     status = online
     有效可用容量 = max_concurrency - current_concurrency > 0

2. 【估算】未指定 workerCount 时，按场景配置公式计算 N；
           已指定时直接使用 N = workerCount

3. 【排序】对候选节点按负载评分升序排列，优先选择资源充裕的节点：
     score = cpu_usage × 0.6 + mem_usage × 0.4
     score 越小越优先

4. 【降级判断】若候选节点数 M < N：
     不拒绝启动，允许降级运行
     实际并发 / RPS 按比例缩减：actual_target = target × (M / N)
     在 task_run 中记录 warning_msg 说明降级情况
     前端初始化面板展示降级告警（橙色 warn 步骤）

5. 【原子锁定】对选中的 M 台节点执行原子锁定，防止并发调度抢占：
     Redis SETNX worker:{workerId}:locked:{runId} 1 EX 30
     锁定失败的节点视为不可用，重新从候选池补充
     若补充后仍不足 → 按实际锁定数降级运行

6. 【快照记录】将选中节点的 workerId / hostname / ip / cpuCores / maxConcurrency
     写入 task_run.worker_snapshots（执行时快照，后续不随节点状态变更）

7. 【并发分配】将总目标并发 / RPS 按各节点 max_concurrency 比例加权分配给每台 Worker：
     Worker_i 分配并发 = actual_target × (Worker_i.max_concurrency / Sum(selected.max_concurrency))
```

#### 特殊场景处理

| 场景 | 处理策略 |
|------|--------|
| 全部节点 offline | 返回错误码 2003，拒绝启动 |
| 候选节点不足（M < N） | 降级运行，缩减目标并发/RPS，写 warning_msg |
| 运行中节点宕机 | 检测到心跳超时 → 标记 run.status=failed，写 error_msg |
| K8s Pod 重启 IP 漂移 | workerId 为配置写死的稳定 ID，IP 变化只影响当次已运行的 run；<br>新 Pod 重新注册后以新 IP 入池，历史快照保留旧 IP（可追溯） |

## 六、Worker 发压引擎设计

### 5.1 VU 阶梯模式

```
按 steps_json 数组顺序逐阶段执行：
  每个阶段：
    若 ramp_time > 0：在 ramp_time 秒内线性增加 goroutine 至本阶段 concurrent 数（从上阶段值起步）
    若 ramp_time = 0：瞬时增加到本阶段 concurrent 数（第一阶段从 0 起步）
    稳定运行 duration 秒后进入下一阶段
  阶段切换时增量启动新 VU（不停止旧 VU），直到所有阶段结束
```

### 5.2 定速 RPS 模式

```
使用 golang.org/x/time/rate 令牌桶限速
rate.NewLimiter(rate.Limit(rps), burst)
每次调用 Default() 前执行 limiter.Wait(ctx)
goroutine 数 = min(rps, max_concurrency)

固定速率子模式（rps_mode=1）：
  全程以 rps 为目标维持恒定速率

阶梯爬升子模式（rps_mode=2）：
  按 steps_json 数组顺序逐阶段执行，每阶段含爬坡 + 稳定两段：
    爬坡段（ramp_time > 0）：从上一阶段 RPS 线性递增到本阶段目标 RPS
      每隔 100ms 重建一次 Limiter，rate = prevRps + (targetRps-prevRps) * elapsed/ramp_time
    稳定段（duration）：以目标 RPS 恒速注入，持续 duration 秒
```

### 5.4 脚本加载机制（Go Plugin）

```
CI 编译产物已上传对象存储，Master 下发任务时携带 artifact_url：

Worker 收到 SubTaskConfig 后：
  1. 检查本地缓存：/var/cache/stress-worker/scripts/{name}/{commit_hash}.so
     命中缓存 → 直接跳到步骤 3
  2. 从对象存储下载 .so：
     GET s3://stress-scripts/scripts/{name}/{commit_hash}.so
     保存到本地缓存目录
  3. plugin.Open(".so")
  4. plugin.Lookup("Script") → 断言为 spec.ScriptEntry 接口
  5. 调度引擎按模式执行 Setup / Default / Teardown

注意：Worker 不执行 go build，编译在 CI 侧完成，CI 与 Worker 使用相同 Docker 镜像
      确保 Go 版本、OS、依赖一致（Go Plugin 跨环境不兼容）

lang 字段扩展预留：
  "go" → 当前支持，CI 编译 .so，Worker plugin.Open 执行
  "js" → 后续扩展，goja 引擎解释执行（无需编译，直接存储脚本内容）
```

### 5.5 RunContext 注入

```
每个 VU goroutine 持有独立的 RunContext：
  RunContext{
    VUId:      goroutine 编号（从 1 开始）
    Iteration: 本 VU 已执行迭代次数
    WorkerID:  本节点 ID
    Vars:      goroutine 级别 KV 存储（VU 间隔离）
    SetupData: Setup() 返回的共享数据（只读）
    HTTP:      sdk_impl.HTTPClient（含耗时统计，自动上报 metrics）
    Check:     sdk_impl.Checker（断言失败标记 fail，不 panic）
    Log:       sdk_impl.Logger（日志上报到 Master 日志流）
    Sleep:     sdk_impl.Sleeper（可被停止信号中断）
  }
```

### 5.6 指标采集结构

```go
type MetricsCollector struct {
    // 原子操作计数
    totalCount   int64
    successCount int64
    failCount    int64
    // RT 直方图（HDR Histogram，用于计算 P50/P90/P99）
    histogram    *hdrhistogram.Histogram
    // 错误分布（按 ScriptError.Code 聚合）
    // key: "business:10001" 或 "system:TIMEOUT", value: int64 计数
    errorStats   sync.Map
    // 当前并发数
    concurrency  int32

    // 接口级指标（按 URL pattern 分桶）
    apiMetrics   sync.Map // key: pattern string, value: *APIMetricsBucket
}

// APIMetricsBucket 单个接口的指标桶（并发安全）
type APIMetricsBucket struct {
    pattern      string
    totalCount   int64
    failCount    int64
    histogram    *hdrhistogram.Histogram
    // 每秒 RPS / RT 时序（Worker 侧保留，用于每秒上报接口级时序）
    secondSeries []APISecondPoint // 按秒追加
}

// APISecondPoint 每秒一个接口的指标快照
type APISecondPoint struct {
    Timestamp  int64   // Unix 秒级时间戳
    RPS        float64 // 该接口在该秒内的 RPS
    AvgRT      float64 // 该接口在该秒内的平均响应时间（ms）
    ErrorRate  float64 // 该接口在该秒内的错误率（0~1）
}
```

> **每秒上报策略**：Worker 每秒同时上报两类指标：
> 1. 全局维度（`POST /internal/metrics/report`）：RPS、RT 分位值、错误计数、并发数
> 2. 接口级时序（`POST /internal/metrics/api-second`）：每个 URL pattern 当秒的 RPS / RT / 错误率
>
> 接口级每秒时序由 Master 写入 Redis（TTL 24h），前端轮询时一并返回，用于报告中接口维度折线图展示。
> 接口级 histogram（分位值计算用）仍在 Run 结束时一次性上报（`POST /internal/metrics/api-report`），流程不变。

### Nacos 配置（`config/nacos.go`，Master 和 Worker 共用结构）

trpc-go 无官方 Nacos 插件，通过 `nacos-sdk-go v2` 直接集成，不走 trpc plugin 体系。
配置结构由应用层自己定义，在启动时初始化 Nacos Client。

```go
// config/nacos.go
type NacosConfig struct {
    Host      string `yaml:"host"`       // Nacos 地址，如 127.0.0.1（开发）/ nacos-svc（K8s）
    Port      uint64 `yaml:"port"`       // 默认 8848
    Namespace string `yaml:"namespace"`  // 命名空间，默认 ""（public）
    Group     string `yaml:"group"`      // 服务分组，默认 DEFAULT_GROUP
}
```

Worker 启动时读取 `POD_IP`（K8s Downward API）或本机 IP，调用 `registry/nacos.go` 完成注册；
Master 启动时调用 `infra/nacos/worker_discovery.go` 订阅 `stress-worker` 服务实例列表。

## 九、部署结构

> 部署分两阶段演进，业务代码完全一致，差异仅在基础设施层，**切换时无需改动任何 Go 代码**。

### 9.1 阶段一：开发 / 测试环境

前端直连 Master，不引入 APISIX，降低开发期复杂度。
Nacos 从第一天就运行（Worker 服务注册是核心功能，不可推迟）。

```
前端（浏览器）
  │ HTTP（localhost 或固定 IP）
  ▼
Master（8080 对外 / 8081 内部）
  │ tRPC（从 Nacos 取 Worker IP，点对点下发）
  ├──► Worker 1（9090）
  ├──► Worker 2（9090）
  └──► Worker N（9090）
       │ 注册/存活心跳（nacos-sdk-go v2）
       ▼
     Nacos（8848）◄── Master 订阅 stress-worker 服务

存储：MySQL（5.7+）、Redis（6.0+）本地启动
```

**Docker Compose 快速启动**（开发环境推荐）：

```yaml
# docker-compose.yml
services:
  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: stress123
      MYSQL_DATABASE: stress
    ports: ["3306:3306"]

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]

  nacos:
    image: nacos/nacos-server:v2.3.0
    environment:
      MODE: standalone
      PREFER_HOST_MODE: hostname
    ports: ["8848:8848", "9848:9848"]
```

Master 和 Worker 本地 `go run` 启动，config 里 Nacos 地址写 `127.0.0.1:8848`。

---

### 9.2 阶段二：生产环境（腾讯云 TKE + 托管中间件）

> 基础设施组件（MySQL、Redis、Nacos）统一使用**腾讯云托管服务**，不部署在 K8s 内。
> 注册中心是其他服务的启动前置依赖，若与业务服务部署在同一 K8s 集群，集群异常时存在循环依赖风险。
> 托管服务在 VPC 内网与 TKE 直连，无需额外网络配置。

| 组件 | 腾讯云托管服务 | 说明 |
|------|--------------|------|
| MySQL | TencentDB for MySQL（CDB） | VPC 内网访问 |
| Redis | TencentDB for Redis | VPC 内网访问 |
| Nacos | TSE 微服务引擎（Nacos 版） | 3节点高可用，原生兼容 Nacos API |
| K8s | TKE（腾讯云容器服务） | Master/Worker 以 Deployment 运行 |

```
外部流量
  │
  ▼
APISIX Ingress Controller（TKE 内，固定域名 api.example.com）
  │  /api/* → master-svc:8080（K8s ClusterIP Service）
  ▼
Master Deployment（TKE Pod）
  │ tRPC（从 TSE 取 Worker Pod IP，点对点下发，绕过 K8s Service）
  ├──► Worker Pod 1（Pod IP:9090）
  └──► Worker Pod N（Pod IP:9090）
       │ 注册时写入 Pod IP（K8s Downward API 环境变量 POD_IP）
       ▼
     腾讯云 TSE（VPC 内网）◄── Master 订阅实例列表

Worker → master-internal-svc:8081（K8s ClusterIP DNS）← 心跳/指标上报

存储层（VPC 内网，TKE 外）：
  CDB MySQL  ←→  Master
  Redis      ←→  Master
```

**Worker Deployment 关键配置**（注入 Pod IP）：

```yaml
# worker-deployment.yaml（片段）
containers:
  - name: stress-worker
    image: stress-worker:latest
    env:
      - name: POD_IP                    # Downward API 注入本 Pod IP，注册到 Nacos/TSE
        valueFrom:
          fieldRef:
            fieldPath: status.podIP
      - name: NACOS_HOST
        value: "your-tse-nacos.internal.tencentcloud.com"   # TSE 内网地址
      - name: NACOS_PORT
        value: "8848"
      - name: MASTER_INTERNAL_ADDR
        value: "master-internal-svc:8081"
    ports:
      - containerPort: 9090
```

**APISIX 路由配置**：

```yaml
apiVersion: apisix.apache.org/v2
kind: ApisixRoute
metadata:
  name: stress-master-route
spec:
  http:
  - name: master-api
    match:
      hosts: ["api.example.com"]
      paths: ["/api/*"]
    backends:
    - serviceName: master-svc
      servicePort: 8080
    plugins:
    - name: limit-req
      enable: true
      config:
        rate: 200
        burst: 100
        key: remote_addr
```

### 9.3 两阶段对比

| | 阶段一（开发） | 阶段二（生产） |
|---|---|---|
| 前端入口 | Master IP 直连 | APISIX（固定域名） |
| JWT 鉴权 | Master 中间件 | Master 中间件（APISIX 插件可选叠加） |
| MySQL / Redis | Docker Compose 本地 | 腾讯云 CDB / Redis（VPC 内网） |
| Nacos | Docker Compose 单机 | 腾讯云 TSE（3节点高可用） |
| Worker IP 来源 | 本机 IP / 容器 IP | K8s Downward API 注入 Pod IP |
| 启动方式 | `go run` + `docker-compose up` | `kubectl apply`（TKE） |
| 基础设施复杂度 | 低 | 中（中间件全托管，无需自运维） |
| 业务代码变更 | — | **零**（纯基础设施配置切换） |
