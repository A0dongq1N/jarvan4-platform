# 压力测试平台 — 概要设计

## 一、整体架构

```
┌─────────────────────────────────────────────────────────┐
│                      前端 (Vue3 + TS)                    │
│   任务管理 │ 场景配置 │ 实时看板(轮询) │ 报告 │ 系统配置  │
└───────────────────────┬─────────────────────────────────┘
                        │ HTTP REST (JSON)
┌───────────────────────▼─────────────────────────────────┐
│               后端 Master 服务 (trpc-go)                  │
│                                                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────────┐  │
│  │ 用户认证  │ │ 任务管理  │ │ 调度引擎  │ │ 数据聚合   │  │
│  └──────────┘ └──────────┘ └────┬─────┘ └─────▲─────┘  │
│                                 │ 下发任务       │ 上报数据 │
└─────────────────────────────────┼───────────────┼────────┘
                        ┌─────────┘               │ tRPC（负载心跳/指标）
         tRPC           │
    ┌─────▼──────┐ ┌────▼───────┐          ┌──Worker N─┐
    │  Worker 1  │ │  Worker 2  │  ···      │ (发压引擎) │
    │ (发压引擎)  │ │ (发压引擎)  │           └──────────┘
    └─────┬──────┘ └────┬───────┘                 │
          │             │                         │
          └──────┬──────┘─────────────────────────┘
                 │ 注册/注销（Nacos Client SDK）
    ┌────────────▼────────────────────────────────┐
    │              Nacos（注册中心）                │
    │   service: stress-worker                     │
    │   Master 订阅实例列表，感知节点上下线           │
    └─────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                       存储层                              │
│        MySQL（持久化）        Redis（实时/状态缓存）        │
└─────────────────────────────────────────────────────────┘
```

## 二、模块划分

### 2.1 后端模块（Master 服务）

| 模块 | 职责 |
|------|------|
| **用户认证模块** | 账号密码登录、手机号短信验证码登录、JWT 签发与验证、权限校验 Filter |
| **任务管理模块** | 任务 CRUD、场景配置的持久化 |
| **调度模块** | 启动/停止任务、Worker 调度（估算所需节点数、负载评分选取、降级告警）、将任务拆分分发给 Worker、管理执行生命周期 |
| **节点管理模块** | Worker 注册、心跳维持、节点状态管理、节点上下线 |
| **数据聚合模块** | 接收 Worker 上报的原始指标、聚合计算、写入 Redis |
| **报告模块** | 压测结束后从 Redis/MySQL 汇总数据生成报告（含脚本快照和参与节点快照）、报告 CRUD、对比、导出 |
| **告警模块** | 轮询指标是否超阈值、触发 Webhook/邮件通知 |
| **审计日志模块** | 记录关键操作到 MySQL |

### 2.2 Worker 服务（独立进程）

| 模块 | 职责 |
|------|------|
| **注册/发现模块** | 启动时向 Nacos 注册（服务名 `stress-worker`，metadata 携带静态配置）；进程退出时主动注销；Nacos Client 负责存活心跳 |
| **任务接收模块** | 接收 Master 下发的子任务配置（SubTaskConfig，含脚本内容） |
| **脚本加载模块** | 接收脚本源码 → go build 编译为 .so → plugin.Open 动态加载 |
| **发压引擎** | 按场景模式（VU 阶梯/RPS）调度 VU goroutine 执行脚本 Default() |
| **SDK 实现层** | 为脚本提供 HTTPClient / Checker / VarStore / Logger 的 Worker 侧实现 |
| **指标采集模块** | 统计 QPS、RT 分位值、成功/失败数，每秒汇总一次 |
| **数据上报模块** | 每秒 POST 聚合指标到 Master（tRPC）；每 5 秒 POST 负载心跳到 Master（tRPC） |

### 2.3 前端模块

| 模块 | 说明 |
|------|------|
| **项目管理** | 登录落地页：项目列表（卡片网格）、新建/删除项目、当前项目切换（TopBar chip） |
| **任务中心** | 任务列表、创建/编辑/复制/删除 |
| **场景配置** | 压测模式（VU 阶梯 / 定速 RPS）、并发/RPS 参数、阶梯 steps、总时长、请求超时、熔断保护 |
| **执行控制** | 启动/停止、定时调度 |
| **实时看板** | 每秒轮询接口，展示 QPS/RT/成功率/并发数/错误分布图表 |
| **压测报告** | 报告列表、报告详情、报告对比、导出 |
| **环境管理** | 环境/全局变量配置 |
| **节点管理** | Worker 节点列表、状态、资源用量（全局共享，不属于某个项目） |
| **用户管理** | 登录、用户 CRUD、角色配置 |
| **告警配置** | 阈值规则、通知渠道 |

## 三、分布式压测设计

### 3.1 Master-Worker 通信方式

```
Master ──── tRPC ────► Worker   (下发：启动/停止子任务)
Worker ──── tRPC ────► Master   (上报：负载心跳 + 实时指标)
Master ──── tRPC ────► Worker   (拉取：指标快照，兜底补偿)

Worker ──── Nacos SDK ────► Nacos   (注册/注销/存活心跳)
Master ──── Nacos SDK ────► Nacos   (订阅 stress-worker 实例列表)
```

- **服务发现（Nacos 负责）**：Worker 启动时向 Nacos 注册服务 `stress-worker`，metadata 携带 `cpu_cores`、`mem_total_gb`、`max_concurrency`、`port`；Master 订阅该服务，Nacos 实例上线/下线事件触发 Master 同步 `worker_node` 表
- **负载心跳（业务层面）**：Worker 每 **5秒** 向 Master POST 负载快照（cpu_usage、mem_usage、current_concurrency、running_run_id），供调度评分用；与 Nacos 存活心跳职责分离
- Worker 每 **1秒** 向 Master POST 一次聚合指标（Push 模式）
- Master 每 **5秒** 主动拉取 Worker 最新快照（Pull 模式，兜底）

### 3.2 任务拆分与 Worker 调度策略

```
用户配置：峰值并发 1000，3 个可用 Worker（各 maxConcurrency=500）

Master 调度流程：
  1. 估算所需 Worker 数 N = ⌈1000 / 500⌉ = 2
  2. 按负载评分（cpu×0.6 + mem×0.4）升序，取前 2 台
  3. 不足 N 台时降级运行，按实际节点比例缩减目标并发，写告警信息
  4. Redis SETNX 原子锁定，防止并发调度抢占
  5. 快照 Worker 信息写入 task_run.worker_snapshots

并发分配（按 maxConcurrency 比例加权）：
  Worker1 (maxConcurrency=500) → 500 并发
  Worker2 (maxConcurrency=500) → 500 并发

定速 RPS 模式（目标 3000 RPS，3 个 Worker）：
  单 Worker 估算额定 RPS = maxConcurrency × 2（按预估 RT=500ms）
  所需 Worker 数 N = ⌈3000 / (500×2)⌉ = 3
  每个 Worker 承担 1000 RPS
```

### 3.3 执行状态流转

```
PENDING ──► RUNNING ──► SUCCESS
    │            │
    │            ▼
    │         STOPPED（手动停止）
    │            │
    │            ▼
    └──────► FAILED（Worker 全部离线等异常）
                 │
                 ▼
          CIRCUIT_BROKEN（熔断自动停止）
```

## 四、数据存储设计

### 4.1 MySQL — 持久化数据

| 表 | 内容 |
|----|------|
| `user` | 用户信息、角色、手机号 |
| `sms_code` | 短信验证码（phone、code、过期时间、已使用标记；生产环境可替换为 Redis） |
| `project` | 项目信息（名称、描述、统计缓存） |
| `task` | 压测任务基本信息（归属 project） |
| `scene_config` | 场景配置（压测模式、并发/RPS参数、阶梯 steps_json、时长、超时等）与 task 1:1 |
| `script` | 压测脚本（名称、语言类型、最新 commit hash、对象存储 artifact_url，归属 project） |
| `script_version` | 脚本版本历史（每次 CI 发布记录一条，commit hash + artifact_url） |
| `task_script` | 任务与脚本的绑定关系（角色、权重；不锁版本，始终使用最新发布版） |
| `task_run` | 每次执行记录（状态、脚本版本快照、参与节点快照、初始化步骤进度、降级告警信息） |
| `report` | 压测报告汇总数据（含脚本快照和参与节点快照） |
| `report_metrics` | 报告的时间序列指标（每秒一条，用于图表回放） |
| `report_api_stats` | 报告接口维度统计（P50~P99、错误率，用于接口维度指标表格） |
| `worker_node` | Worker 节点注册信息（全局共享，不属于项目） |
| `environment` | 环境配置（BASE_URL、全局变量等） |
| `alert_rule` | 告警规则 |
| `audit_log` | 操作审计日志 |

### 4.2 Redis — 实时数据

| Key 规则 | 内容 | TTL |
|----------|------|-----|
| `run:{runId}:status` | 当前执行状态 | 24h |
| `run:{runId}:metrics:latest` | 最新一秒聚合指标（前端轮询用） | 24h |
| `run:{runId}:metrics:ts` | 时间序列指标列表（ZSet，score=时间戳） | 24h |
| `worker:{workerId}:load` | Worker 负载快照（cpu_usage/mem_usage/concurrent），来自业务心跳 | 15s 自动过期 |
| `worker:{workerId}:locked:{runId}` | Worker 调度锁（防并发抢占） | 30s 自动过期 |
| `sms:{phone}` | 短信验证码（6位数字） | 300s（5分钟） |
| `sms:{phone}:lock` | 短信发送冷却锁（防刷） | 60s 自动过期 |

## 五、关键技术方案

| 问题 | 方案 |
|------|------|
| Worker 服务注册与发现 | Worker 通过 `nacos-sdk-go v2` 向 Nacos 注册服务 `stress-worker`，metadata 携带静态配置（cpu_cores、max_concurrency 等）；Master 订阅实例列表，上下线事件同步写 `worker_node` 表 |
| Worker 负载同步 | Worker 每 5 秒向 Master POST 负载心跳（cpu_usage/mem_usage/current_concurrency），与 Nacos 存活心跳职责分离 |
| Worker 实时指标上报 | Worker 每秒通过 tRPC 调用 Master `/internal/metrics/report`，Master 聚合写 Redis |
| 前端实时数据获取 | 前端每 **1秒** 轮询 `GET /api/executions/{id}/metrics`（HTTP） |
| 分布式任务协调 | Master 统一调度，Redis 记录任务状态，避免重复下发 |
| Worker 发压并发 | 每个 Worker 用 goroutine pool 控制并发数，ticker 控制 RPS |
| VU 阶梯模式 | 按 `steps_json` 数组逐阶段启动对应数量 VU goroutine，阶段间增量追加不停止旧 VU |
| RPS 阶梯爬升 | 每 100ms 重建一次令牌桶 Limiter，rate 按 `ramp_time` 线性插值，爬坡结束后进入稳定段 |
| Worker 调度 | 从 Nacos 获取在线实例列表，按负载评分（cpu×0.6 + mem×0.4）选取最优节点；不足时降级运行并写告警；Redis SETNX 防并发抢占 |
| 执行快照 | 启动压测时将脚本版本（commit hash）和参与节点（hostname/ip）快照写入 task_run，冗余到 report，独立于运行时状态变更 |
| 认证 | 支持账号密码（bcrypt）和手机号短信验证码两种方式，均签发 JWT Token；短信验证码存 Redis，TTL 5分钟，60秒防刷锁 |
| 跨域 | trpc-go 中间件统一注入 CORS Header |
| Push/Pull 双模式 | Push 为主路径（Worker→Master，tRPC），Pull 每 5 秒兜底补偿（Master→Worker，tRPC）|
| 脚本分发 | 脚本在独立 Git 仓库开发，CI 编译为 `.so` 上传对象存储（MinIO/OSS），Worker 按 artifact_url 拉取并本地缓存，不在 Worker 侧编译 |
| 脚本编译一致性 | CI 与 Worker 使用相同 Docker 镜像，确保 Go Plugin `.so` 版本兼容 |

## 六、部署结构

> 部署分两个阶段演进，**开发阶段**优先保持简单，**生产阶段**再引入 APISIX 网关。
> 两阶段的业务代码完全一致，差异仅在基础设施层，切换时无需改动任何 Go 代码。

### 阶段一：开发 / 测试环境（Docker Compose 或单机 K8s）

前端直连 Master，Master IP 固定，**不引入 APISIX**，降低开发期复杂度。

```
前端（浏览器）
  │ HTTP（固定 IP 或 localhost）
  ▼
Master（8080 对外 / 8081 内部）
  │ tRPC（下发任务，直连 Pod IP）
  ├──► Worker 1（9090）
  ├──► Worker 2（9090）
  └──► Worker N（9090）
       │ 注册/存活心跳
       ▼
     Nacos（8848）◄── Master 订阅

存储：MySQL + Redis（本地 Docker 或同机部署）
```

**Nacos 在此阶段已存在**：Worker 服务注册/发现是核心功能，开发期就需要验证。
APISIX 不存在，Master 的 JWT 鉴权中间件承担全部认证工作。

---

### 阶段二：生产环境（腾讯云 TKE + 托管中间件）

> 基础设施组件统一使用**腾讯云托管服务**，不部署在 K8s 内。
> 注册中心是其他服务的启动前置依赖，若部署在同一 K8s 集群内，集群异常时存在循环依赖风险；托管服务在 VPC 内网与 TKE 直连，规避此问题并消除自运维负担。

| 组件 | 腾讯云托管服务 |
|------|--------------|
| MySQL | TencentDB for MySQL（CDB） |
| Redis | TencentDB for Redis |
| Nacos | TSE 微服务引擎（Nacos 版，3节点高可用） |
| K8s | TKE（腾讯云容器服务） |

```
外部流量
  │
  ▼
APISIX Ingress Controller（TKE 内，固定域名 api.example.com）
  │  /api/*  →  master-svc:8080（K8s ClusterIP Service）
  │  插件：SSL 终止、限流、可观测性（JWT 鉴权可选下沉至此）
  ▼
Master Pod(s)（TKE Deployment）
  │ tRPC（从 TSE 取 Worker Pod IP，点对点下发，绕过 K8s Service）
  ├──► Worker Pod 1（Pod IP:9090）
  └──► Worker Pod N（Pod IP:9090）
       │ 注册时写入 Pod IP（K8s Downward API 环境变量 POD_IP）
       ▼
     腾讯云 TSE（VPC 内网）◄── Master 订阅实例列表

  Worker → master-internal-svc:8081（K8s ClusterIP DNS）← 心跳/指标上报

存储层（VPC 内网，TKE 外）：CDB MySQL、TencentDB Redis
```

**关键设计决策**：
- Master 的 JWT 鉴权中间件**保留**（纵深防御），APISIX 鉴权插件可选叠加
- Worker → Master 走 K8s Service DNS，不经过 APISIX（内部流量无需网关）
- Master → Worker 走 Pod IP 直连（TSE 提供），绕过 K8s Service 负载均衡（需要点对点路由）

---

### 两阶段对比

| | 阶段一（开发） | 阶段二（生产） |
|---|---|---|
| 前端入口 | Master IP 直连 | APISIX（固定域名） |
| JWT 鉴权 | Master 中间件 | Master 中间件（+ APISIX 可选） |
| MySQL / Redis | Docker Compose 本地 | 腾讯云 CDB / Redis（VPC 内网） |
| Nacos | Docker Compose 单机 | 腾讯云 TSE（3节点高可用） |
| Worker IP 来源 | 本机 IP / 容器 IP | K8s Downward API 注入 Pod IP |
| 基础设施复杂度 | 低 | 中（中间件全托管，无需自运维） |
| 代码变更 | — | 零（纯基础设施切换） |

---

**端口规划**：
- Master 对外：**8080**（前端 API，生产由 APISIX 代理）
- Master 对内：**8081**（Worker 通信：心跳、指标上报）
- Worker：**9090**（接收 Master 下发指令）
- Nacos：**8848**（服务注册与发现）
- APISIX：**80/443**（生产环境对外，开发阶段不启动）
